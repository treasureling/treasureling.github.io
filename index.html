session和cookie

发表于2017-03-16 | 分类于 http协议 |

cookie和session区别

  1. session在用户关闭浏览器后，会话结束，就会消失。客户端的cookie存活的时间更久一点
  2. cookie是存在浏览器的，根据请求的路径自动发送的，服务器端可以对其进行处理。session是存在服务器的，用于区分会话和不同用户的访问
  3. session 的运行依赖 session id，而 session id 是存在 cookie 中的，也就是说，如果浏览器禁用了 cookie ，同时 session 也会失效（但是可以通过其它方式实现，比如在 url 中传递 session_id）
  4. cookie不是很安全，别人可以分析存放在本地的COOKIE并进行COOKIE欺骗
    考虑到安全应当使用session。
  5. Cookie只能保存字符串对象,session可以保存各种类型对象
  6. cookie存储内容过多，会严重占用服务器端带宽，影响性能。session一定时间内保存在服务器上,当访问增多，占用服务器性能

cookie机制

  1. 浏览器检查所有存储的cookie，则把该cookie附在请求资源的HTTP请求头上发送给服务器。
  2. 会话cookie：若不设置过期时间，则表示这个cookie的生命期为浏览器会话期间，关闭浏览器窗口，cookie就消失。
  3. 若设置了过期时间，浏览器就会把cookie保存到硬盘上，关闭后再次打开浏览器，这些cookie仍然有效直到超过设定的过期时间
  4. 只能保存字符串对象，不能保存对象类型
  5. 需要客户端浏览器的支持：客户端可以不支持，浏览器用户可能会禁用Cookie

session机制

  1. session机制是一种服务器端的机制。
  2. 当程序需要为某个客户端的请求创建一个session时，服务器首先检查这个客户端的请求里是否已包含了一个session id，如果已包含则说明以前已经为此客户端创建过session，服务器就按照session id把这个session检索出来

  3. 检索不到，会新建一个

    • 获取session id技术

      1. Cookie传递，如果浏览器禁用了 cookie，则使用URL重写技术，就是把session id直接附加在URL路径的后面

      2. 表单隐藏字段
        在被传递回客户端之前，在 form 里面加入一个hidden域，设置JSeesionId
        <input type=hidden name=jsessionid value="3948E432F90932A549D34532EE2394" />

      3. URL重写,直接在URL后附加上session id的信息

比较好的做法

  • 将登陆信息等重要信息存放为SESSION
  • 其他信息如果需要保留，可以放在COOKIE中

websocket作用

不支持websocket的浏览器实现一样的功能只能通过”ajax轮询”或者“long poll”
缺点：http协议很被动，服务端不能主动联系客户端，只能有客户端发起

  • WebSocket API最伟大之处在于服务器和客户端可以在给定的时间范围内的任意时刻，相互推送信息。
  • WebSocket并不限于以Ajax(或XHR)方式通信，因为Ajax技术需要客户端发起请求，而WebSocket服务器和客户端可以彼此相互推送信息；
  • XHR受到域的限制，而WebSocket允许跨域通信。

http协议

请求篇

由三部分组成：请求行+请求头部+请求数据

  • 请求方法
    • GET 请求获取Request-URI所标识的资源,请求的数据会附在URL之后,传输数据就会受到URL长度的 限制
    • POST 请求服务器接受附在请求后面的数据，一般用作表单提交,数据被包含在请求体中
    • HEAD 请求获取由Request-URI所标识的资源的响应消息报头
    • PUT 请求服务器存储一个资源，并用Request-URI作为其标识
    • DELETE 请求服务器删除Request-URI所标识的资源
    • TRACE 请求服务器回送收到的请求信息，主要用于测试或诊断
    • CONNECT 保留将来使用
    • OPTIONS 请求查询服务器的性能，或者查询与资源相关的选项和需求

  • 请求报头
    举例：

    第一部分：请求行，第一行明了是get请求，以及http1.1版本。
    第二部分：请求头部，第二行至第八行。
    第三部分：空行，第九行的空行。
    第四部分：请求数据，第十行。

响应篇

三部分组成：状态行+消息报头+响应正文

  • 状态码

    • 1xx：指示信息–表示请求已接收，继续处理
    • 2xx：成功–表示请求已被成功接收、理解、接受
    • 3xx：重定向–要完成请求必须进行更进一步的操作
    • 4xx：客户端错误–请求有语法错误或请求无法实现
    • 5xx：服务器端错误–服务器未能实现合法的请求

移动端真机调试

IOS 移动端 原生方法（Safari开发者工具）

  • 手机端：设置 → Safari → 高级 → Web 检查器 → 开。
  • mac端：Safari → 偏好设置 → 高级 → 在菜单栏中显示“开发”菜单

安卓移动端原生方法

  • 首先确保手机上和PC机上装有最新版本的chrome浏览器。http://www.cnblogs.com/meitian/p/5030978.html
  • 手机用数据线连接到电脑上,手机上设置允许通过USB调试,即开发者模式
  • PC的chrome访问chrome://inspect（访问网址后在手机上会弹出一个确认弹框，点击确认即可，如果不确认的），如果连接成功，就会在chrome下显示出对应的手机的型号，如下图
  • 在手机上的chrome浏览器内输入要查看页面元素的网址，比如www.baidu.com
  • 此时PC上的chrome://inspect页面下会出现可以检查的页面
  • 点击对应页签下的inspect，另弹出的chrome页面上可以用来调试操作了

代理工具调试方法

通过代理，这种其实就是移动设备上访问电脑可以访问的内容

  • 启动Charles或者 Fiddler代理抓包工具，查看有关请求和响应、做本地资源映射
  • 本地启动服务，可以依靠工具或者自己用nodejs写一个
  • 手机上设置好代理，保存
  • 手机上打开网页即可在 Charles 中看到代理的请求

Weinre 布局调试

Weinre 是一个相当简单好用的调试工具。它会在你本地创建一个监听服务器，并提供一个 JavaScript，你只需要在需要测试的页面中加载这段 JS，就可以被 Weinre 监听到，在 Inspect 面板中调试你这个页面。

  • npm install -g weinre
  • Mac 在终端执行 ipconfig getifaddr en0 命令/Win 在命令行执行 ipconfig 命令
  • 根据上面命令拿到的ip比如10.189.249.254 终端执行weinre –boundHost 10.189.249.254
  • 这时提供了有一个网址，浏览器打开。提供了一个Target Script。
  • 我们需要把这个 JS 放到我们要调试的页面中，这样访问页面的时候，加载这个 JS，就会被 Weinre 监听到进行控制。

缺点：没有JavaScript debug,可以调试布局问题
优点：浏览器兼容性强

spy-debugger工具，抓包+ 调试

weinre升级版。一站式页面调试、抓包工具。远程调试任何手机浏览器页面，任何手机移动端webview（如：微信，HybirdApp等）。支持HTTP/HTTPS，无需USB连接设备。内部集成了weinre、node-mitmproxy、AnyProxy

  • npm install spy-debugger -g
  • 手机和PC保持在同一网络下
  • 命令行输入spy-debugger，按命令行提示用浏览器打开相应地址
  • 设置手机的HTTP代理，代理IP地址设置为PC的IP地址，端口为spy-debugger的启动端口(默认端口：9888)。
  • 手机安装证书。注：手机必须先设置完代理后再通过(非微信)手机浏览器访问http://s.xxx(地址二维码)安装证书（手机首次调试需要安装证书，已安装了证书的手机无需重复安装)
  • 用手机浏览器访问你要调试的页面即可。

性能优化要点

发表于2016-02-17 | 分类于 性能优化 |

首屏渲染优化

  1. 网络层

    • HTTP2/SPDY
    • HTTPDNS
      • httpdns是面向无线端的域名解析服务，与传统走UDP协议的DNS不同，httpdns基于HTTP协议，基于HTTP的域名解析，减少域名解析部分的时间并解决DNS劫持的问题。

  2. 加载策略

    • 懒加载
    • 非首屏的资源必须延迟加载
    • 首屏不可见内容(通常是交互后才出现的)资源可延迟加载
    • 首屏次要资源（不在首屏视觉重心的内容）可考虑延迟加载
    • 按需加载
    • 前端在onload里同步执行的js、加载的资源都会被计入到pageLoad内，如果其中有一段阻塞性的js，也会完全阻塞pageLoad的触发，规避方案是把onload后的内容放入setTimeout，哪怕是0ms

  3. 图片大小控制

    • 选择合适的图片格式
    • 非透明图尽量使用jpg格式
    • 透明图使用png图时，可使用tinypng进行压缩
    • 支持的端下使用webp格式
    • 合并小图片为sprite图并适当base64化

  4. 减少请求数，尽量combo请求

    • 首屏的数据接口尽可能合并成

  5. 减少资源大小

    • css，js压缩
    • 保持良好的代码习惯，尽量少直接dom操作

  6. 异步加载js

    • async 和 defer 属性，将在onload 事件之前完成。可以同时使用 async 和 defer，这样IE 4之后的所有 IE 都支持异步加载。

      • defer属性ie4就支持了， 所有的defer 脚本保证是按顺序依次执行的。

      • asyncHTML5新增，不能保证脚本会按顺序执行。大部分浏览器支持ie9以上支持

    • 页面中script标签内，用 js 创建一个 script 元素并插入到 document 中

      • 方法一：立即执行异步加载，这种加载方式在加载执行完之前会阻止 onload 事件的触发

      • 方法二：在 onload 时才开始异步加载。这样就解决了阻塞 onload 事件触发的问题。

  7. js模块化管理
    异步加载，需要将所有 js 内容按模块化的方式来切分组织，其中就存在依赖关系，而异步加载不保证执行顺序。RequireJS 、 CommonJS 、seadjs，amd规范

  8. 预加载技术
    需要客户端配合:用户访问H5之前，将页面静态资源（HTML/JS/CSS/IMG…）打包预加载到客户端；用户访问H5时，将网络IO拦截并替换为本地文件IO；从而实现H5加载性能的大幅度提升的方案

    整体性能优化

    1. 借用成熟框架，比如react，vue利用虚拟dom方式，采用diff算法去做视图更新，避免dom操作，很大程度优化用户体验
    2. native化

跨域标准解决方案

fetch(ajax)接口的跨域解决方案 — 跨源资源共享(cors)

优点：解决异步接口请求过程中的各种跨域问题, 替代老的jsonp方案或者iframe跨域方案，从而带来更好的用户体验与开发体验
原理：就是使用自定义的HTTP头部让浏览器与服务器进行沟通，从而决定请求或响应是应该成功，还是应该失败

web客户端

  • 正常的通过fetch或者ajax请求数据，不需要额外设置

  • 如果请求需要带上cookie,需要请求头带上：withCredentials

  • 未避免cors出现先导请求(诸如：options请求)，导致发两次请求，降低性能；content-Type必须为application/x-www-form-urlencoded, multipart/form-data, 或text/plain中的一种

服务端

  • 服务端响应头中必须包含：

  • 如果请求带了cookie，且服务端允许跨站访问携带cookie，响应头必须包含

  • 如果允许web客户端读取服务器的头信息,需要设置Access-Control-Expose-Headers的白名单

iframe 跨域问题 - postMessage方案

  • 父页面发出message
  • iframe页面接收message

form异步提交的跨域方案：formdata + cors方案

css经典布局

经典布局

  1. 弹性盒子布局

    • 要素
      • display:flex
      • flex-direction: row | row-reverse | column | column-reverse
      • flex-wrap:wrap| nowrap| wrap-reverse 换行方式
      • align-items: flex-start | flex-end | center | baseline | stretch
      • justify-content: flex-start | flex-end | center | space-between | space-around
      • flex: auto | initial | none | inherit | [ flex-grow ] || [ flex-shrink ] || [ flex-basis ] flex 属性用于指定弹性子元素如何分配空间。

  2. 圣杯布局

    • 左中右布局，左边和右边固定宽度，中间宽度自适应,需要用到浮动，负边距，相对定位（相对于自身定位relative）

  3. 双飞翼布局

    • 左中右布局，左边和右边固定宽度，中间宽度自适应,需要用到浮动，负边距

  4. 头尾固定高度中间高度自适应布局

    • 兼容ie6的做法利用boxsizing改变盒子模型IE6/IE7下，的box-sizing默认值本就是border-box（注：IE7有一点点不正常，overflow:hidden后神志有所恢复）。
    • 不兼容ie6做法position

  5. 两列布局，左边固定，右边自适应的三种方法

    • 使用margin负值，使右边容器与左边同一行，并100%宽。再在右边容器中放一个子容器，设置margin-left值为左边容器的宽度。原理和双飞翼布局一样
    • 使用position，右边容器必须设置right

  6. 栅格布局：在一个有限的、固定的平面上，用水平线和垂直线（虚拟的线，“参考线”），将平面划分成有规律的一系列“格子”（虚拟的格子），并依托这些格子、或以格子的边线为基准线，来进行有规律的版面布局。

    • 原理：Flowline的宽度为W，column的宽度为c，Gutter的宽度为g，Margin的宽度为m，栅格列数为N

    W = cN + g(N-1) + 2m；g的宽度通常为m的两倍，所以：

    W = (c+g) * N；把c+g记为C，得：

    W = C * N;

    大部分的栅格系统都是此公式的变体。

    • Bootstrap的栅格系统,12列

    • Grid 100切分 百分比布局

every():

some():
