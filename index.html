Node应用程序有两种缓存的处理方式

  • 等到所有数据接收完毕，一次性从缓存读取，这就是传统的读取文件的方式；

session和cookie

发表于2017-03-16 | 分类于 http协议 |

cookie和session区别

  1. session在用户关闭浏览器后，会话结束，就会消失。客户端的cookie存活的时间更久一点
  2. cookie是存在浏览器的，根据请求的路径自动发送的，服务器端可以对其进行处理。session是存在服务器的，用于区分会话和不同用户的访问
  3. session 的运行依赖 session id，而 session id 是存在 cookie 中的，也就是说，如果浏览器禁用了 cookie ，同时 session 也会失效（但是可以通过其它方式实现，比如在 url 中传递 session_id）
  4. cookie不是很安全，别人可以分析存放在本地的COOKIE并进行COOKIE欺骗
    考虑到安全应当使用session。
  5. Cookie只能保存字符串对象,session可以保存各种类型对象
  6. cookie存储内容过多，会严重占用服务器端带宽，影响性能。session一定时间内保存在服务器上,当访问增多，占用服务器性能

cookie机制

  1. 浏览器检查所有存储的cookie，则把该cookie附在请求资源的HTTP请求头上发送给服务器。
  2. 会话cookie：若不设置过期时间，则表示这个cookie的生命期为浏览器会话期间，关闭浏览器窗口，cookie就消失。
  3. 若设置了过期时间，浏览器就会把cookie保存到硬盘上，关闭后再次打开浏览器，这些cookie仍然有效直到超过设定的过期时间
  4. 只能保存字符串对象，不能保存对象类型
  5. 需要客户端浏览器的支持：客户端可以不支持，浏览器用户可能会禁用Cookie

session机制

  1. session机制是一种服务器端的机制。
  2. 当程序需要为某个客户端的请求创建一个session时，服务器首先检查这个客户端的请求里是否已包含了一个session id，如果已包含则说明以前已经为此客户端创建过session，服务器就按照session id把这个session检索出来

  3. 检索不到，会新建一个

    • 获取session id技术

      1. Cookie传递，如果浏览器禁用了 cookie，则使用URL重写技术，就是把session id直接附加在URL路径的后面

      2. 表单隐藏字段
        在被传递回客户端之前，在 form 里面加入一个hidden域，设置JSeesionId

      3. URL重写,直接在URL后附加上session id的信息

比较好的做法

  • 将登陆信息等重要信息存放为SESSION
  • 其他信息如果需要保留，可以放在COOKIE中

react生命周期小结

发表于2017-02-23 | 分类于 react |

组件的生命周期分成三个状态：

Mounting：已插入真实 DOM

  • componentWillMount()

    服务器端和客户端都只调用一次，在初始化渲染执行之前立刻调用。

  • componentDidMount()

    在初始化渲染执行之后立刻调用一次，仅客户端有效（服务器端不会调用）。
    此时可以通过 this.getDOMNode() 来获取相应 DOM 节点。

Updating：正在被重新渲染

  • componentWillUpdate(object nextProps, object nextState)

    在接收到新的props 或者 state 之前立刻调用。在初始化渲染的时候该方法不会被调用。

  • componentWillReceiveProps

    可以作为react 在 prop 传入之后， render() 渲染之前更新 state 的机会

  • shouldComponentUpdate

    在接收到新的props 或者 state，将要渲染之前调用。该方法在初始化渲染的时候不会调用，在使用 forceUpdate 方法的时候也不会。

    • componentDidUpdate(object prevProps, object prevState)

      在组件的更新已经同步到DOM 中之后立刻被调用。该方法不会在初始化渲染的时候调用。
      使用该方法可以在组件更新之后操作DOM 元素。

Unmounting：已移出真实 DOM

  • componentWillUnmount()
    在组件从DOM 中移除的时候立刻被调用。

websocket作用

不支持websocket的浏览器实现一样的功能只能通过”ajax轮询”或者“long poll”
缺点：http协议很被动，服务端不能主动联系客户端，只能有客户端发起

  • WebSocket API最伟大之处在于服务器和客户端可以在给定的时间范围内的任意时刻，相互推送信息。
  • WebSocket并不限于以Ajax(或XHR)方式通信，因为Ajax技术需要客户端发起请求，而WebSocket服务器和客户端可以彼此相互推送信息；
  • XHR受到域的限制，而WebSocket允许跨域通信。

gulp和webpack理解

发表于2016-11-08 | 分类于 前端工程化模块化 |

gulp

gulp是一个基于流的构建工具，可以自动执行指定的任务，简洁且高效。利用 Node.js 流的威力，你可以快速构建项目并减少频繁的 IO 操作。

  • 基本原理
    • 使用 gulp.src 读入文件转为 stream（数据流）的形式
    • 再通过 .pipe() 接口对其进行链式加工处理。.pipe 可以视为流的“管道/通道”方法，任何类型的流都会有这个 .pipe 方法去成对处理流的输入与输出。中间是一些中间步骤，每一步都对数据流进行一些处理
    • dest方法将数据流写入文件

  • 举例

  • task()

    • task方法用于定义具体的任务

    • 指定并发执行一组任务，每个任务是异步调用

    • 按次序运行，可以把前一个任务写成后一个任务的依赖模块。例如，css任务依赖greet任务，所以css一定会在greet运行完成后再运行

    • default，就表明它是“默认任务”，在命令行直接输入gulp命令，就会运行该任务。

  • watch()监视文件变化，一旦变化执行回调方法

webpack

配置

  • plugins 是插件项:用来拓展Webpack功能的，它们会在整个构建过程中生效，执行相关的任务
  • entry 是页面入口文件配置，output 是对应输出项配置

  • loaders 是最关键的一块配置。它告知 webpack 每一种文件都需要使用什么加载器来处理：

    • test：一个匹配loaders所处理的文件的拓展名的正则表达式（必须）
    • loader：loader的名称（必须）
    • include/exclude:手动添加必须处理的文件（文件夹）或屏蔽不需要处理的文件（文件夹）（可选）；
    • query：为loaders提供额外的设置选项（可选）

  • resolve 主要是分析依赖时的路径识别的配置

  • externals 是配置外部依赖的，如果使用某些第三方库的时候，就可以通过这个配置来识别外部第三方库的依赖

  • devserver:使用webpack构建本地服务器

    • contentBase:默认webpack-dev-server会为根文件夹提供本地服务器，如果想为另外一个目录下的文件提供本地服务器，应该在这里设置其所在目录（本例设置到“public”目录）
    • port:设置默认监听端口，如果省略，默认为”8080“
    • inline:设置为true，当源文件改变时会自动刷新页面
    • colors:设置为true，使终端输出的文件为彩色的
    • historyApiFallback:在开发单页应用时非常有用，它依赖于HTML5 history API，如果设置为true，所有的跳转将指向index.html

  • 热插拔插件：Hot Module Replacement

    • 效果：它允许你在修改组件代码后，自动刷新实时预览修改后的效果
    • 步骤：
    • 在webpack配置文件中添加HMR插件；
    • 在Webpack Dev Server中添加“hot”参数；
  • devtool:
    • 注释：
      其实就是选择一种调试代码的方式:生成Source Maps（使调试更容易）
    • 开发总是离不开调试，如果可以更加方便的调试当然就能提高开发效率，不过打包后的文件有时候你是不容易找到出错了的地方对应的源代码的位置的，Source Maps就是来帮我们解决这个问题的。通过简单的配置后，Webpack在打包时可以为我们生成的source maps，这为我们提供了一种对应编译文件和源文件的方法，使得编译后的代码可读性更高，也更容易调试。

      gulp & webpack结合一起使用

      分工

    • gulp：处理html压缩/预处理/条件编译，图片压缩，精灵图自动合并等任务
    • webpack：管理模块化，构建js/css。

模块化小结

发表于2016-09-11 | 分类于 前端工程化模块化 |

CommonJS规范

  • 原理：
    • 服务器端的 Node.js 遵循 CommonJS规范，该规范的核心思想是允许模块通过 require 方法来同步加载所要依赖的其他模块，然后通过 exports 或 module.exports 来导出需要暴露的接口。
    • 模块输出的是被输出值的拷贝，一旦输出一个值，模块内部的变化就影响不到这个值
  • 优点：
    • 服务器端模块便于重用
    • NPM 中已经有将近20万个可以使用模块包

  • 缺点：

    • 因为所有的模块都存放在本地硬盘，可以同步加载完成，等待时间就是硬盘的读取时间。但是，对于浏览器，这却是一个大问题，因为模块都放在服务器端，等待时间取决于网速的快慢，可能要等很长时间，浏览器处于”假死”状态,阻塞程序运行。所以更适合服务端。

  • 实现：

    • 服务器端的 Node.js
    • Browserify
    • webpack

      AMD 规范

  • 原理：主要接口 define(id?, dependencies?, factory)，它要在声明模块的时候指定所有的依赖 dependencies，并且还要当做形参传到 factory 中，对于依赖的模块提前执行，依赖前置。

  • 优点：

    • 适合在浏览器环境中异步加载模块
    • 可以并行加载多个模块
  • 缺点：
    • 提高了开发成本，代码的阅读和书写比较困难，模块定义方式的语义不顺畅
  • 实现：
    • requireJS
    • curl
    • webpack 。webpack 来说，我们可以直接在上面书写 commonJS 形式的语法，无须任何 define （毕竟最终模块都打包在一起，webpack 也会最终自动加上自己的加载器

CMD规范

  • 原理：规范和 AMD 很相似，尽量保持简单，并与 CommonJS 和 Node.js 的 Modules 规范保持了很大的兼容性。

  • 优点：

    • 依赖就近，延迟执行
    • 可以很容易在 Node.js 中运行
  • 缺点：
    • 依赖 SPM 打包，模块的加载逻辑偏重

ES6 模块

  • 原理：
    • ES6 模块的设计思想，是尽量的静态化，使得编译时就能确定模块的依赖关系，以及输入和输出的变量。CommonJS 和 AMD 模块，都只能在运行时确定这些东西。
    • es6模块加载模块输出的是值的引用，是动态引用，并且不会缓存值
  • 优点：
    • 容易进行静态分析
    • 面向未来的 EcmaScript 标准
  • 缺点
    • 原生浏览器端还没有实现该标准
    • 新版的 Node.js才支持
  • 实现：Babel
